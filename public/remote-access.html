Remote Access Guide

How to securely access your hub dashboard from outside the LAN

When Do You Need This?

By default, session-forge hub runs on your local network (LAN). Developers connect over http://192.168.x.x:3700. This is the most secure setup because nothing leaves your network.

You might want remote access if:

  • You have remote developers who aren't on the office LAN
  • You want to check the dashboard from home
  • Your team works across multiple offices connected by VPN
Important: Exposing the hub to the internet means your data is accessible remotely. Always use HTTPS and consider IP whitelisting or VPN access.

Option 1: VPN (Recommended)

The safest approach. Remote users connect to your office VPN, then access the hub as if they were on the LAN. No changes needed to the hub server.

1
Set up a VPN server

WireGuard, OpenVPN, or Tailscale (easiest). Your IT admin likely already has one.

2
Remote users connect to VPN

Once connected, they're on the same network as the hub.

3
Access the hub normally

Same URL: http://192.168.x.x:3700

Pros: No ports exposed to the internet. The hub stays completely internal. Zero configuration changes needed.

Option 2: Reverse Proxy with SSL

If VPN isn't practical, you can expose the hub through a reverse proxy with HTTPS. This is how most web apps handle it.

1
Point a subdomain to your server

e.g. forge.yourcompany.com → your server's public IP

2
Install Nginx and Certbot 
sudo apt install nginx certbot python3-certbot-nginx
3
Create Nginx config 
# /etc/nginx/sites-available/forge-hub
server {
    listen 80;
    server_name forge.yourcompany.com;

    location / {
        proxy_pass http://127.0.0.1:3700;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
4
Enable and get SSL certificate 
sudo ln -s /etc/nginx/sites-available/forge-hub /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
sudo certbot --nginx -d forge.yourcompany.com
5
Restrict access (recommended)

Add IP whitelisting to only allow known office/home IPs:

# Add inside the location block:
allow 203.0.113.0/24;    # Office IP range
allow 198.51.100.5;       # Admin home IP
deny all;

Option 3: Cloudflare Tunnel (Zero Trust)

No ports opened, no public IP needed. Cloudflare tunnels the connection through their network. Free tier available.

1
Install cloudflared 
# Linux
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64 -o /usr/local/bin/cloudflared
chmod +x /usr/local/bin/cloudflared

# Windows (winget)
winget install cloudflare.cloudflared
2
Create the tunnel 
cloudflared tunnel login
cloudflared tunnel create forge-hub
cloudflared tunnel route dns forge-hub forge.yourcompany.com
3
Run the tunnel 
cloudflared tunnel --url http://localhost:3700 run forge-hub
Bonus: Cloudflare Access lets you add email-based authentication on top, so only approved team members can reach the dashboard. No extra code needed.

Configuring Reporters for Remote Access

Once your hub is accessible remotely, update each developer's reporter config to use the public URL:

# ~/.session-forge-reporter.json
{
  "hubUrl": "https://forge.yourcompany.com",
  "agentApiKey": "sfh_agent_..."
}

Or use the CLI flag:

npx session-forge-reporter --hub https://forge.yourcompany.com --key sfh_agent_...
Note: All API traffic is authenticated with bearer tokens. However, always use HTTPS when going over the internet to protect the tokens in transit.
