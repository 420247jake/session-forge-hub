Why Your AI Session Data Should Stay on YOUR Network

session-forge hub is built on the principle that your AI development context is too valuable and too sensitive to leave your machine.

📊 What data does session-forge track?

Every time your AI coding agent works on a task, session-forge captures structured context about the session:

  • Session checkpoints — snapshots of what the agent is working on, files touched, and current progress
  • Architectural decisions — what was chosen, what alternatives existed, and why
  • Dead ends — what was tried, why it failed, and what lesson was learned
  • Journal entries — session summaries including breakthroughs, frustrations, and collaboration notes
  • User profiles — developer preferences, project lists, and working patterns
  • File paths and code context — which files are being modified and why

🔒 Why this data is sensitive

This is not telemetry or usage metrics. This is intelligence about your codebase:

  • Proprietary architecture — decisions reveal how your system is designed and why
  • Vulnerability history — dead ends often expose security-sensitive debugging paths
  • Business logic — session context captures the intent behind code, not just the code itself
  • Team workflows — collaboration notes and preferences reveal how your team operates
  • File system structure — paths expose your project organization and infrastructure layout

If this data were exposed to a third party, they would have a detailed map of your software's architecture, known weaknesses, and development priorities.

⚠ The problem with cloud-hosted alternatives

Cloud-hosted AI session management tools typically:

  • Route your architectural decisions through third-party servers
  • Store your debugging history in someone else's database
  • Require internet access to function, creating a dependency on external services
  • Aggregate data across users, creating attractive targets for attackers
  • Include analytics, tracking pixels, or telemetry that phones home
  • Subject your data to the provider's privacy policy, which can change at any time

Even with encryption, the metadata alone (which projects you are working on, when, and how often) is valuable intelligence that you should not be sharing.

🏠 How session-forge hub is different

session-forge hub is designed from the ground up to be local-first:

  • Runs on your machine or your LAN — the server binary runs wherever you choose
  • Zero external API calls — the hub never contacts any external server, period
  • No analytics or tracking — no Google Analytics, no Mixpanel, no Segment, nothing
  • No phone-home mechanism — no update checks, no license validation, no heartbeats
  • Flat JSON storage — your data sits in plain JSON files on your filesystem, fully inspectable
  • Open source — you can audit every line of code to verify these claims
  • No account required — no sign-up, no email, no OAuth, just a local API key
  • Optional Stripe only — the only external service is Stripe for donations, and it is completely optional

🔐 The "No External Network Calls" Guarantee

session-forge hub makes zero outbound network requests. It does not check for updates, send telemetry, validate licenses, or communicate with any server outside your network. The only network activity is receiving inbound requests from your own agents on your own network. You can verify this by running it behind a firewall that blocks all outbound traffic — it will work perfectly.

Your AI session data is a map of your software's brain. Keep it close.
